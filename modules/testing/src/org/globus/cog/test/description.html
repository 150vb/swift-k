The Testing Framework Description

1. Design:

The present framework allows the user to mainly perform 5 different tests for testing the working of the GridFTP in Java CoG Kit. They are explained below.

Version Test:

This test retrieves the version of a gridFTP server through a GRAM gatekeeper. The gridFTP server and the gatekeeper must be on the same machine in order to function properly. The test may fail

  • if there is no GRAM server installed or running properly
  • if there is no Grid FTP server installed or running properly

Ping Test:

This test tries to ping to a gridFTP server and established connection if successful.It verifies if the user has an account to access the server. The test may fail

  • mso-list:l4 level1 lfo2;tab-stops:list .5in">if there is no Grid FTP server installed or running properly
  • if the user does not have access to the server running properly
  • if there are network problems connecting to the host machine

List Test:

This test checks list function is supported in the gridftp It tries to connect to the remote machine and retrive list of files from the user's home directory. The test may fail

  • if Ping Test already fails
  • if the server has a file or directory with format not supportedby the current Java CoG Kit Package.

2Party Test:

This test verifies both the client to server and server to client file transfers.It creates a dummy file, adds some data to it and transfers to the remote machine. It then gets back the same file. The test may fail

  • if the user does not have proper access to the server.
  • if the Grid FTP server version is not compatible with one supported by the Java CoG Kit package.

3Party Test:

This test tests for third party transfer support in the machine.It tries to transfer the file obtained from second party to itself. The test may fail

  • if the user does not have proper access to the server.
  • if the Grid FTP server version is not compatible with one supported by the Java CoG Kit package.
  • if Grid FTP server does not support third party transfers.

Report :

The output of the tests is shown in a table format in results.html file. Each cell in the table reports the success or failure of the test. If it it a failure, by clicking on the link the user get the detailed stacktrace of the place where the error occured.

2. Using the Testing Framework:

Installation Instructions

1. Checkout the jglobus,ogce and ftp packages from the cvs.

>cvs -d :pserver:anonymous@pitcairn.mcs.anl.gov:/home/dsl/cog/CVS login

Press enter for password. 

>cvs -d :pserver:anonymous@pitcairn.mcs.anl.gov:/home/dsl/cog/CVS co jglobus

>cvs -d :pserver:anonymous@pitcairn.mcs.anl.gov:/home/dsl/cog/CVS co ogce

>cvs -d :pserver:anonymous@pitcairn.mcs.anl.gov:/home/dsl/cog/CVS co ftp

 

2. Go to ogce directory and say

 > ant dist

 Running Instructions:

 1. Change to ogce directory.

 2. First run the Java CoG setup component to make sure your ip address

is correct and certificates are loaded

 >ant -f demos.xml setup

 3. Create a proxy using the following command.

 >ant -f demos.xml login

 4. The names of the machines to be tested are given in a text file

named "machines.txt" present in the ogce directory.

Edit this file to add the machines you need to test.

 5. Run the test using the command.

 >ant -f test.xml ftp

 

3.Output:

The testing framework generates the output in a file named "results.html" in

a tabular format. The names of the machines are the rows and the different tests

are the columns. Each cell indicates the test result. If a test succeeds it prints

out an "OK" message. Otherwise it prints out "Failed" message which has a hyperlink to

the detailed error message.

 Note: The test may take a long time if you have many machines to test. You can

open the "results.html" file and see the intermediate results. Refresh the page to

see the latest results.
